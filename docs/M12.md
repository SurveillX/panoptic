# M12 — Expanded Bucket-Marker Derivation

Panoptic derives event markers from finalized detection buckets. M12 expands the marker vocabulary from two (`spike`, `after_hours`) to five, adding `drop`, `start`, and `late_start`. `underperforming` is implemented but held back until it can be validated on broader historical data (see [§ Deferred: `underperforming`](#deferred-underperforming)).

## Marker catalog

| Marker key        | Event type          | What it means                                                | Ships in M12 |
|-------------------|---------------------|--------------------------------------------------------------|--------------|
| `spike`           | `activity_spike`    | Anomaly detector fired on a per-fragment basis                | yes (pre-M12) |
| `after_hours`     | `after_hours_activity` | Detections in the night window (UTC 21:00–06:00)         | yes (pre-M12) |
| `drop`            | `activity_drop`     | Daytime bucket collapses below baseline                      | **yes**       |
| `start`           | `activity_start`    | First meaningful activity of the UTC day after a quiet stretch | **yes**     |
| `late_start`      | `late_start`        | Today's first active bucket is ≥ 2h past the camera's typical hour | **yes** |
| `underperforming` | `underperforming`   | Active but well below baseline during the work window        | deferred      |

All markers share the shape persisted in `panoptic_buckets.event_markers` and mapped by `shared/events/build.build_event_row_from_bucket_marker` into `panoptic_events`:
```json
{"ts": "iso8601", "event_type": "str", "label": "str", "confidence": 0.0-1.0}
```

## Data flow

Fragment-based markers (`spike`, `after_hours`) derive in `shared/signals/derive.derive_markers` during `transform_to_bucket_record`. History-based markers (`drop`, `start`, `late_start`, and `underperforming` when enabled) derive in `shared/signals/derive.derive_history_markers` called from `ingest_bucket` after `BucketHistory` is fetched from the DB. The two marker lists merge into `event_markers` before the INSERT, deduped on `(event_type, ts)` for idempotent replay.

`BucketHistory` (see [shared/signals/history.py](../shared/signals/history.py)) captures three separate baselines used by different markers:

- **Rolling-bucket** (last 96 buckets ≈ 24h): drives `drop` and `underperforming` thresholds.
- **Same-day context**: first-active-of-day gate for `start` and `late_start`; warm-up skip for `underperforming`.
- **Day-level** (last 14 calendar days): typical first-active hour + count of days with real activity. Drives `late_start`.

Each query uses strict-past-only conditions (`bucket_start_utc < :target`) so re-derivation over historical buckets never leaks future data.

## Thresholds (tuning knobs)

All thresholds live in [shared/signals/derive.py](../shared/signals/derive.py) as named constants. Consumers see marker labels, not numbers — change one file to tune.

### `drop`

Fires when a previously active camera collapses below baseline during daytime hours.
- `_DROP_MIN_ROLLING_SAMPLE = 16` — ≥ 4h of history.
- `_DROP_MIN_ROLLING_MEAN = 5.0` — exclude always-dead cameras.
- `_DROP_SIGMA = 2.0` — current < mean − 2σ.
- Threshold must itself be ≥ 1 — when `mean − 2σ < 1` the distribution is too bursty for a below-baseline rule to be meaningful; suppress rather than fire. This is why yard cameras (std > mean) don't trigger drop, and is intentional: 2σ-below-mean isn't the right signal for bursty surveillance traffic.
- Daytime gate `[06:00, 21:00)` UTC — `after_hours` owns night.
- Confidence: `clamp((mean − current) / (std + 1), 0, 1)` — typically saturates to 1.0 once drop fires; the signal is binary-ish.

### `start`

Fires on the first active bucket of the UTC day after a sustained-quiet tail.
- `_START_MIN_QUIET_MINUTES = 120` — ≥ 2 hours of quiet before the transition. Absorbs natural short lulls (coffee breaks, brief passes).
- `_START_MIN_DETECTIONS = 5` — skip tree-shadow / one-person-walks-by noise.
- First-of-day gate: at most one `start` per camera per UTC day.
- Confidence: fixed 0.9 (low-entropy event; severity gradient isn't meaningful).

Quiet-run timing is computed from `EXTRACT(EPOCH FROM :target - MAX(bucket_end_utc))` where the aggregate walks the most recent *active* bucket. Time-based rather than count-based so gaps in bucket emission (trailer offline, sparse test fixtures) correctly count as quiet.

### `late_start`

Fires when today's first active bucket is ≥ 2h past the camera's typical first-active hour.
- `_LATE_START_MIN_DAYS_WITH_ACTIVITY = 5` — stable 14-day baseline required (not just "calendar days looked at", but "days where this camera had real activity").
- `_LATE_START_HOUR_DELAY_THRESHOLD = 2` — two hours late before we call it late.
- `_LATE_START_MIN_DETECTIONS = 5`.
- Confidence: `clamp((hour_delay − 2) / 6, 0.5, 1.0)`. 2–5h late flattens at 0.5; ramps to 1.0 by 8h.
- **Intentionally co-fires with `start`** on the same bucket when both guards pass — the summary agent's dormant branches treat them independently.

## UTC-day handling — v1 compromise

Day-level comparisons (`first_active_bucket_start_today`, `typical_first_active_hour_utc`) are **UTC-based in M12**. Consequences:

- For US-Eastern sites (today's deployed yard trailer), UTC 14:00 ≈ local 09:00 — consistent, reasonable.
- For West-Coast sites, DST transitions shift the UTC-day boundary twice a year.
- For sites in distant time zones, UTC-day boundary effects can produce edge-case markers near local midnight.

This is an **explicit v1 compromise**, not a footnote. Adding per-site time zones requires either a per-serial config or a fleet-registry change, both out of M12 scope. The failure mode is edge-case "odd-looking" markers around midnight local, not systematic incorrectness. **M12.1 candidate: per-serial time-zone handling.**

## Mode-1 rederive — evaluate before you ship

[scripts/rederive_markers.py](../scripts/rederive_markers.py) re-derives markers over historical buckets without writing anything to the database. Use it to spot-check new thresholds, audit FP rates, or evaluate a candidate marker before flipping the production gate.

```bash
# Dry-run against a serial over the last 14 days; writes a diff JSONL
# plus a human summary to logs/m12-rederive/<ts>/.
python -m scripts.rederive_markers --serial <SN> --days 14 --out <dir>

# Restrict to a single camera:
python -m scripts.rederive_markers --serial <SN> --camera <CAM_ID> --days 14

# Evaluate only some marker families (defaults to all implemented):
python -m scripts.rederive_markers --serial <SN> --produce drop,start

# Mode 2: append new-family markers to existing event_markers. Dedups
# on (event_type, ts); never deletes or overwrites prior markers.
python -m scripts.rederive_markers --serial <SN> --days 14 --apply-new
```

The summary emits per-marker totals, per-camera histograms for the top 10 by proposed-marker count, and FP-gate checks keyed to plan §5a. Mode 3 (`--overwrite-all`) is a placeholder that refuses to run in M12 — reserved for a future release that retunes `spike` or `after_hours` thresholds.

## Deferred: `underperforming`

Implementation lives in [shared/signals/derive._derive_underperforming](../shared/signals/derive.py) and is callable via `derive_history_markers(..., produce=IMPLEMENTED_HISTORY_MARKERS)`. It is **not in `PRODUCED_HISTORY_MARKERS`**, so production ingest never calls it.

Why deferred: the Mode-1 dry-run against yard data (1108 buckets, 10 cameras, 3 days available at the time) produced **zero** underperforming fires. The yard cameras are bursty (std > mean/2), which — by the same logic that suppresses false `drop` fires — suppresses `underperforming`. With only 3 days of yard data and no low-variance cameras for comparison, there was no signal to validate or reject P12e against the plan's FP-gate (§5a: `underperforming ≤ 4× drop`, with both at zero the ratio is degenerate).

Flipping underperforming to production is a one-line change in `shared/signals/derive.py`:
```python
PRODUCED_HISTORY_MARKERS: frozenset[str] = frozenset({
    MARKER_DROP, MARKER_START, MARKER_LATE_START,
    MARKER_UNDERPERFORMING,   # add after Mode-1 gate passes
})
```

Re-run the Mode-1 dry-run once there's sufficient history depth and/or camera diversity to produce a meaningful fire count, then inspect the per-camera histogram and spot-check ≥ 10 firings for FP rate before flipping.

## Consumer paths (what lights up automatically)

Downstream services react to the new event types without code changes:

- **Summary agent** — [services/panoptic_summary_agent/executor.py:205-231](../services/panoptic_summary_agent/executor.py) already has branches for every M12 marker key. Prompt-count injection at lines 448-476.
- **Search API** — [services/search_api/executor.py:321-323](../services/search_api/executor.py) accepts any `event_type` filter value. Query `?event_type=activity_drop` returns rows as soon as the first fires.
- **Agent layer** — routes through Search API; new markers become citable in answers.
- **Report generation** — [shared/report/prompts.py](../shared/report/prompts.py) passes event items through untyped; new markers appear in reports without template changes.
- **Operator UI** — `.evt-activity_drop`, `.evt-activity_start`, `.evt-late_start` CSS classes added alongside existing `.evt-activity_spike` / `.evt-after_hours_activity` in [services/panoptic_operator_ui/static/style.css](../services/panoptic_operator_ui/static/style.css).

## Replay scenarios

[scripts/replay_buckets.py](../scripts/replay_buckets.py) ships synthetic scenarios for each marker. Each feeds through `ingest_bucket` — no hand-injected markers. Derivation is expected to fire the target marker with no false fires elsewhere.

- `--scenario drop_midday` — 5h of steady daytime activity then one zero-detection midday bucket. Expect one `drop` on the target bucket.
- `--scenario start_of_day` — 3h of pre-dawn quiet then daytime activity. Expect one `start` on the first active bucket.
- `--scenario late_start_by_3h` — 5 days of history at 07:00 UTC; target day starts at 10:00. Expect `start` + `late_start` co-firing on the 10:00 bucket.

## Known limitations (M12 → M12.1)

- Per-serial / per-site time zones (UTC-day compromise, see above).
- `underperforming` ship-gate (needs more diverse data).
- Per-object-type drop / underperforming — today's derivations aggregate across object types.
- Persistent `camera_baseline` table if per-bucket DB load becomes measurable at fleet scale.
- Orphan-event cleanup after threshold changes (old `panoptic_events` rows stay content-addressed and idempotent forever — a data-hygiene tool can land if operators want "as of threshold vNext" views).
